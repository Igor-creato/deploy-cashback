# diag/ — снятие улики prod PHP-FPM USR2 spawn-loop

Tracking: **Igor-creato/deploy-cashback#1**

Цель: системно установить root-cause лупа PHP-FPM на проде после
`docker kill -s USR2 wordpress`, не пересоздавая прод-контейнер до снятия
улики. Это **диагностический этап** (Phase 1 систематической отладки) —
здесь нет фикса; фикс проектируется отдельно после доказанного root-cause.

## Почему так

- Луп **детерминирован** на проде (каждый USR2), сайт остаётся `200`
  (выживший пул держит трафик), само-восстановление за ~секунды. Прошлая
  улика потеряна (контейнер был пересоздан).
- Конфиг стэка **git-идентичен** прод↔staging → различие в **рантайме**.
- Grafana/cAdvisor **не** покажет churn воркеров (process/cpuLoad-метрики
  отключены, php-fpm-exporter удалён). Улика только из `docker logs` +
  `/proc`/`docker top` + FPM full-status.

## Жёсткие правила

- **НЕ** `restart`/recreate контейнер `wordpress`, пока артефакты не сняты.
  Recreate уничтожит улику (как в прошлый раз).
- Никаких `apt-get`/инсталляций в контейнер.
- Единственное мутирующее действие = **ОДИН** `USR2` (штатный graceful,
  санкционирован владельцем). Делать в окно низкого трафика.
- Артефакты (`SUMMARY.txt`, census-отчёты) — оператор **вставляет в чат**.
  Сырьё остаётся на сервере в `OUTDIR`.

## A. ПРОД: захват лупа (оператор)

1. Доставка (как обычно — стэк правится локально, на сервере только pull):
   ```
   cd /opt/cashback/deploy   # путь прод-стэка
   git pull
   ```
2. Окно низкого трафика. Запуск:
   ```
   bash service/scripts/diag/capture-fpm-usr2.sh
   ```
   Скрипт: снимает пред-перепись (Q1) → стартует непрерывные стримы логов
   ДО сигнала → просит подтверждение → шлёт **ровно один** `USR2` →
   снимает съёмку через полное восстановление → строит `SUMMARY.txt`.
   На запрос подтверждения ввести `yes` (любое другое — безопасная отмена).
3. По завершении **вставить в чат содержимое** пути, который скрипт
   напечатал последней строкой: `<OUTDIR>/SUMMARY.txt`
   (например `/tmp/fpm-usr2-<ts>/SUMMARY.txt`).
4. **Контейнер не трогать.** Если Claude попросит конкретный сырой файл —
   он лежит рядом в том же `OUTDIR` (`fpm.log`, `pre-usr2.txt`,
   `samples.tsv`, `events.log`, `post.txt`).

### Тест H1 — USR2 при ЗАНЯТОМ воркере (главный)

Прогон #1 (idle-USR2) дал чистый reload без лупа → баг state-dependent.
Гипотеза H1: луп нужен занятый длинным запросом воркер в момент USR2
(на проде — `as_async_request_queue_runner` во время деплоя). Тест:
```
bash service/scripts/diag/capture-fpm-usr2.sh --hold-seconds 150
```
Harness сам займёт РОВНО один FPM-воркер безобидным `<?php sleep(150);`
(скрипт в **/tmp контейнера**, НЕ в webroot — nginx недостижим, исчезает
при любом recreate; удаляется по выходу), дождётся `active>=1`, затем
пошлёт один USR2. 150с > graceful-окна и > `request_terminate_timeout=120s`
— проверяются и graceful-drain, и SIGKILL-на-120s. Подтвердить `yes`,
вставить `SUMMARY.txt` в чат (там строка «ПРЕДУСЛОВИЕ H1: АКТИВНО»).

Вариант «ловить реальный релиз вместо ручного USR2»:
```
bash service/scripts/diag/capture-fpm-usr2.sh --no-fire   # армируется и ждёт
```
затем запустить штатный деплой плагина в другом окне; скрипт сам поймает
`reloading` в логе. Совместимо с `--hold-seconds`.

Полезные env-переключатели (по умолчанию разумны):
`CONTAINER`, `SAMPLE_INTERVAL` (0.5s), `MAX_CAPTURE_SECONDS` (300),
`STABLE_SECONDS` (30), `OUTDIR`.
